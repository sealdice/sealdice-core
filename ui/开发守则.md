# SealDice UI 迁移开发守则

本文面向把 `sealdice-ui` 迁移到当前 `ui` 项目的开发者。当前项目不是简单复制老前端，而是以 `ui` 的新架构承接老业务能力：保留后端强绑定和 Hash 部署特性，逐步迁移海豹业务页面，同时避免把老项目的大 store、手写请求和页面混杂结构一并带过来。

## 总体原则

- `ui` 是迁移目标，`sealdice-ui` 是业务资产来源。
- 新页面使用 Vue 3 + TypeScript + Naive UI + Tailwind CSS v4。
- 不引入 Element Plus、UnoCSS、文件路由、MSW、自动模块注册或 Axios service/store 模式。
- 不把 `sealdice-ui` 页面整文件复制后“能跑就行”。迁移时要拆清楚：路由页面、业务组件、shared 组件、API、类型、状态。
- 服务器数据优先走 API + Vue Query；Pinia 或 VueUse 只放客户端 UI 状态和少量持久化偏好。
- 本项目当前依赖 `src/api/generated`，如果缺失，需要先生成 OpenAPI 客户端，否则 `type-check` 和 `build` 会失败。

## 目录边界

推荐按以下边界放代码：

```txt
src/
  api/              # OpenAPI 生成客户端与 API client 配置
  components/       # 业务组件（按模块分子目录）
    shared/         # 可跨页面复用的 UI/编辑器/渲染组件
  features/         # 业务模块逻辑、模型、组合式函数
  layouts/          # 页面外壳：default/auth/plain/wide
  pages/            # 路由页面（文件路由自动注册）
  router/           # 手写路由、菜单配置、进度条、路由类型
```

迁移规则：

- `pages/`：路由入口页，使用文件路由自动注册。例如 `pages/mod/deck.vue` → `/mod/deck`。
- `components/shared/`：只放跨多个页面复用、业务耦合较低的组件。
- `features/<domain>/`：放某个业务域的模型、组合式函数、页面私有组件。业务域例子：`connect`、`customText`、`upload`、`theme`。
- 不要继续沿用老项目把页面塞进 `components/` 的习惯。
- 不要新建一个巨大的 `store/index.ts` 承接所有海豹状态。

## 已引入的基础设施

### 布局与路由

本项目使用 `vue-router/auto-routes` 文件路由。`pages/` 下的 `.vue` 文件自动注册为路由：
- `pages/index.vue` → `/`
- `pages/mod/deck.vue` → `/mod/deck`
- `pages/misc/base-setting.vue` → `/misc/base-setting`
- `pages/custom-text/[category].vue` → `/custom-text/:category`

路由元信息通过 `router/index.ts` 的 `routeMeta` 配置，不写在文件里：

```ts
const routeMeta: Record<string, RouteRecordRaw['meta']> = {
  '/mod/deck': { layout: 'default', title: '牌堆管理' },
  '/mod/js': { layout: 'default', title: 'JS 扩展' },
};
```

选择建议：

- 普通管理页：`default`
- 大编辑器、大表格、资源预览、日志/diff：`wide`
- 登录/注册/找回密码：`auth`
- 跳转回调、授权确认、一次性流程页：`plain`

### 菜单配置

侧边栏菜单来自 `router/navigation.ts` 的 `appNavigation`，不是写死在组件里。

菜单项可以配置：

```ts
{
  label: '资源管理',
  routeName: 'resource',
  requiresAuth: true,
}
```

注意：

- `routeName` 要和路由里的 `name` 一致。
- 需要登录才显示的菜单加 `requiresAuth: true`。
- 登录后不该显示的入口加 `guestOnly: true`。
- 不要在 `AppSidebar.vue` 里手写业务菜单。

### VueUse

当前已引入 `@vueuse/core`。使用边界：

- 可以用于 UI 偏好：侧边栏折叠、窗口断点、剪贴板、临时本地状态。
- 可以用于响应式工具：`useBreakpoints`、`useMediaQuery`、`useLocalStorage`。
- 不要用它替代认证流程或服务端数据缓存。
- 不要把后端返回的业务列表长期持久化到 localStorage。

示例：

```ts
const sidebarCollapsed = useLocalStorage('ui.sidebar.collapsed', false);
const isMobile = useMediaQuery('(max-width: 860px)');
```

### NProgress

当前通过 `src/router/progress.ts` 接入 `nprogress`。

约定：

- NProgress 只表示路由导航进度。
- API 请求、保存按钮、列表加载继续使用 Vue Query 的 `isPending` / `isFetching` 或组件局部 loading。
- 不要在页面里手动 `NProgress.start()` / `NProgress.done()`，除非是在扩展路由级行为。

### Shared Components

已经完成前置迁移的 shared 组件：

- `FoldableCard.vue`：可折叠卡片。
- `DiffViewer.vue`：代码 diff 查看器。
- `ResourceRenderer.vue`：图片、音频、视频渲染。
- `ConditionBuilder.vue`：自定义回复条件构建器。
- `NestedRuleEditor.vue`：拖拽嵌套规则编辑器。
- `CustomTextBox.vue`：自定义文案分组容器。

使用规则：

- shared 组件应通过 props / emits / `defineModel` 接收数据，不直接调用 API。
- shared 组件不读取全局路由、不读写 token、不直接依赖某个页面的 Query Key。
- 如果组件只服务一个业务页面，先放到 `features/<domain>/components/`，不要急着放 shared。
- 从 `sealdice-ui` 搬来的 SVG 图标应尽量替换为 Naive UI 现有图标或统一的图标组件；不要在每个按钮里散落大段内联 SVG。

## API 与状态管理

### 推荐方式

服务端数据：

```txt
OpenAPI generated client -> Vue Query -> 页面/业务 composable
```

客户端 UI 状态：

```txt
局部 ref/reactive -> VueUse localStorage/sessionStorage -> 必要时 Pinia
```

迁移老页面时，不要把 `sealdice-ui/src/store/index.ts` 的模式照搬过来。老 store 里混合了：

- 类型定义
- token
- 连接列表
- 骰子配置
- 自定义文案
- 日志
- API 调用
- 页面动作

这些在新项目里要拆开。

### 类型放置

如果类型来自后端 OpenAPI，优先使用生成类型（`src/api/generated/`）。

如果后端暂时没有 OpenAPI 覆盖，临时业务类型放在：

```txt
src/features/<domain>/types.ts
```

不要把大量海豹业务类型都塞进 `src/types.ts` 或某个全局文件。

### Query Key

新增查询时要有稳定 Query Key。建议按业务域分组：

```ts
const diceBaseInfoQueryKey = ['dice', 'base-info'];
const connectionsQueryKey = ['connections', 'list'];
```

更新成功后，只 invalidate 相关 Query，不要粗暴清空整个 QueryClient，除非是退出登录或 token 失效。

## 从 `sealdice-ui` 迁移时的处理方式

### 页面迁移步骤

1. 先识别老页面职责：页面布局、表单状态、API 调用、可复用子组件、纯展示组件。
2. 把路由入口放到 `pages/`（文件路由自动注册）。
3. 把该业务域专用组件放到 `components/<domain>/`。
4. 真正跨业务复用的组件才放到 `components/shared/`。
5. 把 API 调用从页面里抽到 OpenAPI 或 `features/<domain>/api` 的薄封装。
6. 用 Vue Query 管理读取、刷新、保存后的失效。
7. 在 `router/index.ts` 的 `routeMeta` 中配置布局和标题，在 `router/navigation.ts` 中接入菜单。

### 老代码不能直接照搬的部分

不要照搬：

- `sealdice-ui/src/store/index.ts` 的单一大 store。
- `sealdice-ui/src/api/index.ts` 的通用 `request(method, url, data)` 风格。
- 页面里直接写大量 token/localStorage 操作。
- 页面里混合心跳、弹窗、新闻、登录、安全检查、路由跳转等全局逻辑。
- Element Plus 组件和样式类。

可以吸收：

- 真实业务流程和交互细节。
- 老页面的字段含义、接口路径、错误文案。
- 已迁移到 shared 的纯组件能力。
- 测试模式水印、离线提示、高级设置入口等产品逻辑，但要拆成独立 composable 或布局级组件。

## 认证与全局流程

认证状态管理在 `features/auth/`：

- `state.ts` — token 存储与读取
- `useAuthSession.ts` — 登录/登出会话管理
- `crypto.ts` — 密码哈希工具

迁移要求：

- 不要在页面里直接读写 token，通过 `features/auth/state.ts` 处理。
- 退出登录必须清理 token。
- 实时通信使用 `features/realtime/client.ts`。
- 日志流使用 `features/base/logStream.ts`。

## 样式与交互

- 使用 Naive UI 组件，不引入 Element Plus。
- 页面可以使用 Tailwind CSS v4 原子类，但复杂组件内部应保持 scoped CSS 简洁可读。
- 不要照搬老项目的灰色大背景和 Element Plus 默认后台风格；新 UI 应保持当前 `AppShell` 的清晰后台风格。
- 文本不要挤在按钮里；移动端要检查按钮换行、表单列布局、抽屉菜单。
- 大编辑器、diff、资源预览、日志列表优先使用 `wide` 布局。

## 路由与部署

- 当前仍使用 `createWebHashHistory`，不要改成 history 模式。
- 页面地址保持 `/#/...`，这对和后端同源部署更稳。
- 开发代理仍由 Vite 配置处理，不要在页面里写死后端端口。

## 验收命令

提交迁移前至少运行：

```sh
pnpm run lint
pnpm run type-check
pnpm run build-only
```

如果 `type-check` 或 `build-only` 因 `src/api/generated` 缺失失败，先生成客户端：

```sh
pnpm run generate-client
```

如果生成失败，先确认：

- `openapi.json` 是否存在于 `ui/` 根目录。
- 或 `VITE_API_PROXY_TARGET` / `DEV_PROXY_SERVER` / `VITE_API_BASE_URL` 指向的后端是否能返回有效 `/openapi.json`。
- 不要用手写临时 stub 假装通过构建。

开发预览：

```sh
pnpm run dev
```

如果 `5173` 被占用，可以临时运行：

```sh
pnpm exec vite --host 0.0.0.0 --port 5175 --strictPort
```

## 迁移完成标准

一个页面迁移完成，不只是"能打开"。至少要满足：

- 路由在 `router/index.ts` 的 `routeMeta` 中有明确 `layout`。
- 菜单在 `router/navigation.ts` 中正确出现或有意隐藏。
- 服务端数据由 Vue Query 管理。
- 保存、删除、上传等 mutation 后会刷新相关 Query。
- 页面没有直接复制老的大 store 逻辑。
- 没有引入 Element Plus、Axios service/store 新体系或 UnoCSS。
- 桌面和移动端布局都可用。
- `pnpm run lint` 通过。
- `pnpm run type-check` 和 `pnpm run build-only` 通过。
